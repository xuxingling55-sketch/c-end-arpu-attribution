# C 端 ARPU 归因分析 Skill

## 使用边界

本 Skill 用于“取数 + 写作型”的 C 端经营归因，重点回答：

- 为什么某月 C 端 ARPU 环比/同比变化？
- 是付费率问题，还是客单价问题？
- 是用户客群、商业化/电销渠道，还是商品承接导致？
- 是否存在上新、切品、年级迁移导致的商品口径变化？

默认优先做**环比归因**。只有用户明确要求同比，才同时补同比。

## 前置确认

如果用户没有明确时间，必须先反问时间，不要自行默认当前月或最近月。

必须确认：

- 分析月份：例如 `202604`
- 对比月份：默认上月，例如 `202603`；若用户要求同比，则为去年同月
- 指标对象：默认 `C 端 ARPU`
- 是否只看环比：默认只看环比
- 输出形式：默认 Markdown 正文；如用户要求写入文件，再落盘

如果用户只说“4 月”但未给年份，且上下文无法唯一确定年份，先反问年份。

## 标准口径

### 活跃用户

- 主表：`aws.business_active_user_last_14_day`
- 粒度：先按 `month_id + u_user` 去重
- 分母：`count(distinct u_user)`
- 该表是 C 端/私域活跃主表，不用 `dws.topic_user_active_detail_day` 直接作为 ARPU 分母。

### 正价营收

- 订单表：`dws.topic_order_detail`
- 金额：默认 `sum(sub_amount)`
- 订单筛选：
  - `is_test_user = 0`
  - `u_user is not null`
  - `original_amount >= 39`
  - `business_gmv_attribution in ('电销','商业化')`
- 不默认限制 `status = '支付成功'`；只有用户明确要求“支付成功订单”或指定订单状态口径时才加该条件。
- ARPU 分子必须限定在同月 C 端活跃用户池内：订单表必须 join 活跃池后再汇总。

### 核心指标

- `ARPU = 活跃用户池内正价营收 / C端活跃用户数`
- `付费率 = 活跃用户池内正价付费用户数 / C端活跃用户数`
- `ARPPU = 活跃用户池内正价营收 / 活跃用户池内正价付费用户数`
- `AOV = 活跃用户池内正价营收 / 正价订单数`

## 取数执行

优先使用项目脚本取数：

```bash
python3 scripts/run_arpu_attribution.py \
  --analysis-month 202604 \
  --compare-month 202603 \
  --output-dir outputs/c_end_arpu_202604_vs_202603
```

如果未传 `--compare-month`，脚本默认使用分析月上月。

脚本依赖以下环境变量：

- `HIGH_VALUE_SSH_HOST`
- `HIGH_VALUE_SSH_USER`
- `HIGH_VALUE_SSH_PASS`
- `HIGH_VALUE_DB_HOST`
- `HIGH_VALUE_DB_PORT`
- `HIGH_VALUE_DB_USER`
- `HIGH_VALUE_DB_PASS`

脚本输出：

| 文件 | 用途 |
|---|---|
| `core_monthly.csv` | 整体月指标：活跃、付费、订单、营收、ARPU、付费率、ARPPU、AOV |
| `user_status.csv` | 用户客群归因：新增、续费、老未、高净值等 |
| `gmv_channel.csv` | 商业化/电销渠道归因 |
| `product_l3.csv` | 商品二级/三级类目归因 |
| `good_name.csv` | 单品下钻 |
| `strategy_goods.csv` | 规划提分课/全面进阶课等新老商品承接校正 |

写作前必须读取并引用这些 CSV 中的关键结果。若脚本失败，先报告失败原因；不要凭历史结果或记忆写归因。

## 标准取数顺序

1. **整体月指标**：活跃用户、付费用户、订单数、正价营收、ARPU、付费率、ARPPU、AOV。
2. **公式拆解**：`ARPU = 付费率 × ARPPU`，拆出付费率影响和 ARPPU 影响。
3. **用户客群拆解**：按 `business_user_pay_status_statistics_month` 拆。
4. **渠道拆解**：按 `business_gmv_attribution` 拆商业化/电销。
5. **商品类目拆解**：按 `business_good_kind_name_level_2/3` 拆。
6. **单品下钻**：按 `good_name` 找主要变化商品。
7. **业务策略校正**：如果出现老品为 0、新品出现，检查是否有上新、切品、升年级、年级迁移。

## 公式拆解规则

用 `ARPU = 付费率 × ARPPU` 做二因素拆解。

推荐解释口径：

- 付费率影响：`(本期付费率 - 基期付费率) × 基期ARPPU`
- ARPPU 影响：`本期付费率 × (本期ARPPU - 基期ARPPU)`
- 两者合计约等于 ARPU 变化。

判断规则：

- ARPU 降、付费率降、ARPPU 升：转化问题，客单价托底。
- ARPU 降、付费率升、ARPPU 降：客单价/商品结构问题。
- 收入降幅大于活跃降幅：活跃池收入产出能力变弱。
- 活跃降幅大、付费率稳定：流量规模问题更明显。
- 付费率和 ARPPU 同时下降：转化与客单结构共同恶化。

## 渠道归因规则

渠道字段使用 `business_gmv_attribution`，重点看 `商业化` 与 `电销`。

分析时必须同时写清：

- 付费用户变化
- 付费率贡献变化
- ARPPU 变化
- 营收变化
- 对整体 ARPU 贡献变化

解释规则：

- 商业化付费用户下降多：偏流量承接、活动入口、低客单转化问题。
- 商业化 ARPPU 下降：成交结构偏低或高客单承接不足。
- 电销付费用户下降少但营收下降大：高客单线索、销售承接、续费/规划类商品成交问题。
- 电销 ARPPU 上升但营收下降：剩余成交更高客单，但成交人数不足。

不要只写“商业化下降/电销下降”，必须说明两者问题形态是否不同。

## 商品与上新校正规则

商品类目先看 `business_good_kind_name_level_2/3`，再用 `good_name` 下钻。

遇到以下情况，不能直接写“断档”：

- 老商品 4 月为 0
- 新商品 4 月出现
- 商品名出现 `七升八`、`八升九`、`小升初`、`初升高`、`新高二`、`新高三`
- 用户提到“上新策略”“向下买一年”“升年级购买”

正确做法：

1. 把老商品和新商品放在同一张表中对照。
2. 判断新品是否承接老品贡献。
3. 如果新品未完全覆盖老品，写为“切品后的承接不足”，不要写“需求完全断档”。

推荐表达：

> 4 月不是规划/进阶类商品完全没卖，而是商品名从旧年级版切换为升年级商品。新品有承接，但未完全覆盖老品贡献，因此表现为切品后的承接不足。

## 写作要求

- 每张表后必须有“描述性分析”，不能只摆数据。
- 不要混用环比和同比；用户没要求同比时，不写同比。
- 结论先回答“是什么问题”，再回答“发生在哪些客群/渠道/商品”。
- 表格只保留支撑结论的关键数字，避免过多明细淹没主线。
- 对业务因果保持克制：数据只能证明“与…一致 / 由…驱动 / 与…相关”；需要业务验证的写成假设。
- 输出到文件前，先确保章节编号连续、口径一致、没有“断档”等未经校正的绝对表述。

## 常见错误

- 用户没给时间就直接跑当前月。
- 用全量订单营收除以 C 端活跃用户，导致 ARPU 分子分母不一致。
- 未经用户要求就默认添加 `status = '支付成功'`。
- 只看 `good_name` 老品为 0，就写“商品断档”。
- 只摆表格，不解释每个数据意味着什么。
- 把商业化和电销合并写，忽略两者问题形态不同。
- 用户只问环比，却加入同比导致主线混乱。
